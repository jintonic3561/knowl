サイクル挙動

  • docker/Dockerfile + docker/docker-compose.yml で本プロジェクトを常時起動コンテナとして起動 (docker-out-of-docker)
  • docker/entrypoint.sh が cron_interval_minutes から crontab を生成し cron を foreground で実行
  • 毎ティックで uv run knowl run-once --config /etc/knowl/config.yaml --credentials ~/.claude/.credentials.json を起動
  • 使用量を取得しゲート判定 → 余裕なしなら no-op で次回まで待機
  • 余裕あり: open issue 収集 → Claude が優先度判定 → 1 件選定 → 対象 container 上で claude -p (implementation / investigation テンプレ)
  • 結果 JSON (action, summary, url, followups) を Slack サマリへ。limit 到達は ⚠️ アラートで通知

既知の留保点

  • 対象 devcontainer (例: stockbot-dev) を VSCode の Dev Containers 拡張で開いている間、knowl の cycle 実行中に拡張が devcontainer に SIGTERM を送って exit させるケースを観測。entrypoint の trap "exit 0" 15 で正常終了するため docker exec 側は 137 を返す。回避策: knowl 実行中は VSCode を閉じる、または devcontainer.json の shutdownAction: "none" を検討。
  • 対象 devcontainer 内の Claude Code には別途認証が必要 (knowl の credentials は中継しない)。未ログイン時は Not logged in · Please run /login で exit 1。ホスト側 ~/.claude を devcontainer に bind するか手動 claude /login 運用。さらに claude がメインユーザ (vscode 等) の home 配下にしか入っていない場合、knowl.yaml の対象 entry に container.user: vscode を設定して docker exec --user を通すこと。.envrc 経由で GH_TOKEN 等を流している場合は container.exec_prefix: ["direnv", "exec", "."] も併記する (非対話 docker exec では direnv hook が発火しないため)。
  • Claude usage API のレスポンス形状は仕様不確定。knowl.usage.parse_usage_payload は utilization/used_percent/remaining_percent/remaining+limit をフォールバック処理する形で頑健化済。
  • OAuth トークンの自動 refresh は実装せず、期限切れ時は TokenExpiredError で Slack 通知 + 当該サイクル no-op。ホスト側で claude を実行して credentials.json を更新する運用前提 (ディレクトリマウントなので knowl コンテナは再起動不要)。
  • devcontainer kind は docker exec で扱う簡略実装。devcontainers CLI が必要な特殊運用には未対応。
